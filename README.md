# Drift tidy robot — Gazebo simulation (ROS 2)

Home-tidying mobile manipulator simulation: **Gazebo Classic** world, **drift_robot** URDF/Xacro, and a Python **navigator** mission node (waypoints, LiDAR gating, pick/place via link attacher).

**Demo video:** https://drive.google.com/file/d/1vkPUcPmtT00DpefxxOgRN7NGZS4GzAXB/view?usp=sharing

---

## How the stack behaves (read this before judging the sim)

These are **intentional** design choices, not loose bugs:

1. **Arm motion / small drift** — The URDF uses Gazebo **implicit spring dampers** on the arm joints for a stable “V” rest pose. The node **re-sends** joint trajectories periodically so commanded poses are held against those springs. You may see small continuous correction; that is expected.

2. **Stopping short of the box** — On the final return segment, **forward LiDAR** below ~0.40 m triggers an **early stop** so the nose does not drive into the collection box. The robot then runs the **drop** sequence from that pose.

3. **`AttachLink` on `right_forearm_link`** — In Gazebo Classic, **fixed** child links (e.g. gripper palm) are often **merged** into the parent rigid body. The physics engine exposes **`right_forearm_link`** as the weld target at the gripper centre; attaching there is correct. Finger links are still used from **`/link_states`** to judge approach distance during **creep**.

4. **Mission phases** — The node runs a string state machine: **TO_OBJ → REACH_DOWN → CREEP → GRAB_PAUSE → LIFT → TO_BOX → DROP_PAUSE → … → DONE** (see `navigator.py`).

More detail: **`docs/APPROACH.md`**.

## ROS 2 workspace layout

The repository root (the folder created by **`git clone`**, usually named **`drift_tidy_robot`**) contains **`drift_ws/`** at **one level below** that root. The colcon workspace root is **`…/drift_tidy_robot/drift_ws/`** (or **`…/<your-clone-dir>/drift_ws/`** if you cloned under another name).

Packages live under a **real** **`drift_ws/src/`** directory (not a symlink, not `drift_ws/drift_ws/src`).

| Path on disk | Purpose |
|--------------|---------|
| `drift_ws/src/drift_robot` | Main robot package (URDF, world, launch, `navigator.py`) |
| `drift_ws/src/IFRA_LinkAttacher/linkattacher_msgs` | Service definitions for Gazebo link attach/detach |
| `drift_ws/src/IFRA_LinkAttacher/ros2_LinkAttacher` | Gazebo plugin + ROS 2 bridge (IFRA Cranfield). **Note:** folder name uses capitals **`L`** and **`A`**; the ROS package **name** inside `package.xml` is still **`ros2_linkattacher`** (what `colcon` and `ros2 pkg` use). |

**Verify before `rosdep` / `colcon`:**

```bash
cd /path/to/drift_tidy_robot/drift_ws    # use your actual clone path
ls src
# Expect: drift_robot   IFRA_LinkAttacher
ls src/IFRA_LinkAttacher
# Expect: linkattacher_msgs   ros2_LinkAttacher
```

After building, **`drift_ws/install/`** holds the overlay. **`build/`**, **`install/`**, and **`log/`** under `drift_ws/` are generated by colcon (ignored by git).

### `drift_robot` package contents

```
drift_ws/src/drift_robot/
├── urdf/drift_robot.urdf.xacro
├── worlds/home.world
├── launch/simulation.launch.py
├── scripts/navigator.py
├── meshes/
├── CMakeLists.txt
└── package.xml
```

There is **no** `config/` directory and **no** `waypoints.yaml`. Outbound/return poses and the mission object list are **hardcoded** in `scripts/navigator.py` (`_OUTBOUND_PATH`, `_RETURN_PATH`, `OBJECTS`, `BOX_POS`, `DROP_POS`). `simulation.launch.py` does not load waypoint YAML.

## Topics, services, and mission I/O (sanity checklist)

Use this when comparing against `ros2 topic list` / `ros2 service list` with the sim running.

| Name | Type | Direction | Role |
|------|------|-----------|------|
| `/cmd_vel` | `geometry_msgs/Twist` | **Pub** (navigator) | Differential drive command |
| `/odom` | `nav_msgs/Odometry` | **Sub** (navigator), **recorded** (bag) | Pose for waypoints / room logic |
| `/scan` | `sensor_msgs/LaserScan` | **Sub** (navigator), **recorded** (bag) | Forward-window obstacle / box wall |
| `/camera/image_raw` | `sensor_msgs/Image` | **recorded** (bag) | From Gazebo camera plugin |
| `/camera/camera_info` | `sensor_msgs/CameraInfo` | (Gazebo) | Camera calibration |
| `/joint_states` | `sensor_msgs/JointState` | **Pub** (Gazebo plugin) | Wheels + arm + fingers |
| `/arm_trajectory` | `trajectory_msgs/JointTrajectory` | **Pub** (navigator) | Arm + finger targets → Gazebo joint pose plugin |
| `/link_states` | `gazebo_msgs/LinkStates` | **Sub** (navigator) | Finger link poses for creep distance |
| `/nav_done` | `std_msgs/Bool` | **Pub** (navigator) | `true` when mission completes |
| `/ATTACHLINK` | `linkattacher_msgs/srv/AttachLink` | **Client** (navigator) | Weld object to robot link |
| `/DETACHLINK` | `linkattacher_msgs/srv/DetachLink` | **Client** (navigator) | Release object |

TF: **`odom` → `base_footprint`** (and robot model frames) from the front-axle `diff_drive` plugin.

| APT package | Role |
|-------------|------|

## Clone, dependencies, build, launch

1. **`git clone`** creates a new directory (by default the repo name, e.g. **`drift_tidy_robot`**).  
2. Open **`drift_ws/`** inside that directory — that is the colcon workspace root.  
3. Run **`rosdep`** and **`colcon`** from **`drift_ws/`**, not from the parent repo root.

```bash
git clone https://github.com/<your-org>/drift_tidy_robot.git
cd drift_tidy_robot          # top level of the clone (name matches repo by default)
cd drift_ws                  # colcon workspace — must contain ./src/

ls src
cd src
# Expect: drift_robot   IFRA_LinkAttacher
git clone https://github.com/IFRA-Cranfield/IFRA_LinkAttacher.git
cd ..
rosdep install --from-paths src --ignore-src -r -y

source /opt/ros/humble/setup.bash   # or: source /opt/ros/jazzy/setup.bash

colcon build --symlink-install
source install/setup.bash
```

If you cloned into a differently named folder, use **`cd <that-folder>/drift_ws`** instead of **`cd drift_tidy_robot/drift_ws`**.

**Launch** (Gazebo + robot + bag recorder + `navigator` starting ~2 s after spawn):

```bash
# From drift_ws/ with overlay sourced
ros2 launch drift_robot simulation.launch.py
```

```bash
ros2 launch drift_robot simulation.launch.py gui:=false   # headless, if your install supports it
```

## What a successful run looks like (pass / fail baseline)

**Timeline (wall clock is approximate; sim uses `use_sim_time`):**

1. Gazebo opens with **`home.world`**; the robot spawns in room 1.
2. After ~**2 s**, **`navigator`** starts (`spawn_entity` exit handler).
3. The robot drives **toward the doorway** (waypoints), enters **room 2**, and approaches the **cyan tall cylinder** at **(3.2, −1.5)**.
4. The arm **reaches down**, **creeps**, **closes fingers**, then **`AttachLink`** runs; the arm **lifts** and the robot **returns** toward the collection box in the SW of room 1.
5. Near the box, the robot may **stop early** when **LiDAR** sees the box wall (~0.4 m) — then **drop / detach / stow**.
6. Terminal ends with a **`MISSION COMPLETE`** block: objects collected, rooms visited, distance driven; **`/nav_done`** publishes **`true`**.

**Example log lines to expect** (match **`navigator.py`** literally where noted):

- `MissionNode ready — waiting for odometry`
- After ~20 odom messages: `Odom live — start pos (X.XX,Y.YY).` — two floats with **two decimal places** and a **trailing period** (grep example: `Odom live — start pos` or `Odom live` )
- `[MISSION] Outbound → obj ...`
- `[CREEP] ... dist_xy=...` then attach / lift / return
- `MISSION COMPLETE` with `Objects collected : 1/1` (for the default single-object mission)

If **`ATTACHLINK` services are missing**, the world plugin is not loaded — fix `home.world` / **`ros2_LinkAttacher`** package build; the node still logs phases but physics attach will not hold the object.

## Troubleshooting

| Issue | Suggestion |
|-------|------------|
| `rosdep` finds nothing | Run from **`drift_tidy_robot/drift_ws`** and ensure **`ls src`** shows **`drift_robot`** and **`IFRA_LinkAttacher`** |
| `gazebo` not found | Install **`gazebo`** and source `/opt/ros/<distro>/setup.bash` |
| **`ros2_linkattacher` build fails** | On Ubuntu 22.04 install **`libgazebo11-dev`** (not only the virtual `libgazebo-dev` metapackage) |
| `linkattacher_msgs` not found | `colcon build` from **`drift_ws`** with ROS sourced |
| Mission exits immediately | Gazebo up, overlay sourced, check **`ros2 service list \| grep ATTACH`** if pick fails |
| Robot falls through ground | Spawn `z` in `simulation.launch.py`; wheel `kp`/`kd` in URDF |
| `xacro` error | `sudo apt install ros-<distro>-xacro` |
| Navigator idle | `ros2 topic echo /odom` should update after spawn |
